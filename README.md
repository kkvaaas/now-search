# Сейчас ищут

Топ самых популярных поисковых запросов за последние 5 минут

## 1. Локальный запуск и примеры запросов к API

### Запуск

```bash
docker compose up --build
```

После запуска доступны:

```text
HTTP API: http://localhost:8080
NATS: nats://localhost:4222
```

### Проверить, что сервис работает

```bash
curl http://localhost:8080/healthz
```

### Отправить поисковое событие в брокер

```bash
nats pub search.events '{"event_id":"evt-1","query":"iphone","user_id":"user-1","session_id":"session-1"}'
```

### Получить популярные поисковые запросы

```bash
curl "http://localhost:8080/api/v1/trends/top?limit=10"
```

`limit=10` означает, что нужно вернуть не больше 10 запросов.

### Посмотреть stop-list

```bash
curl http://localhost:8080/api/v1/stop-list
```

### Добавить слово в stop-list

```bash
curl -X POST http://localhost:8080/api/v1/stop-list -H "Content-Type: application/json" -d '{"word":"iphone"}'
```

### Удалить слово из stop-list

```bash
curl -X DELETE "http://localhost:8080/api/v1/stop-list?word=iphone"
```

### Добавить поисковое событие без NATS

```bash
curl -X POST http://localhost:8080/debug/search-events -H "Content-Type: application/json" -d '{"query":"iphone","user_id":"user-1"}'
```

### Запустить тесты

```bash
go test ./...
```

## 2. Контракт данных из брокера

Сервис читает сообщения из NATS subject:

```text
search.events
```

Пример payload:

```json
{
  "event_id": "evt-123",
  "query": "iphone 15",
  "user_id": "user-42",
  "session_id": "session-777",
  "created_at": "2026-05-26T15:20:00Z"
}
```

`event_id` — номер события, пригодится для защиты от дублей

`query` — текст поискового запроса, его сервис считает и показывает в топе

`user_id` — id пользователя, нужен, чтобы один пользователь не мог слишком сильно накрутить один и тот же запрос

`session_id` — id сессии, используется вместо `user_id`, если пользователь не авторизован

`created_at` — время, когда пользователь сделал запрос, нужно, чтобы считать топ только за последние 5 минут

## 3. Обоснование архитектуры

Общая схема:

```text
Search Service -> NATS -> now-search -> HTTP API -> Widget
```

Основные части сервиса:

`Aggregator` — хранит поисковые события за последние 5 минут

`TopCache` — хранит заранее посчитанный топ запросов

`StopList` — хранит слова, которые нужно скрывать из топа

`SpamGuard` — защищает топ от простой накрутки

`NATS consumer` — читает события из брокера

`HTTP API` — отдает топ и управляет stop-list

NATS выбран, потому что он просто запускается через Docker и хорошо подходит для передачи событий между сервисами

Данные хранятся в памяти, потому что нам нужен топ только за последние 5 минут.Так проще и быстрее добавлять события и удалять старые данные

Проблема: если сервис перезапустить, текущий топ пропадет, но для виджета актуальных запросов это допустимо


## 4. Trade-offs и бизнес-логика

В задаче сказано, что запросов на чтение топа будет больше, чем входящих событий, поэтому топ пересчитывается не при каждом HTTP-запросе, а заранее. Из-за этого топ может отставать примерно на 1 секунду, но чтение работает быстрее

Топ считается только за последние 5 минут. Для этого каждое событие хранит `created_at`, а старые события удаляются. Минус: данные хранятся в памяти и пропадают при рестарте сервиса

Stop-list сделан динамическим, чтобы можно было скрывать слова без перезапуска сервиса. Сейчас фильтрация идет по вхождению строки

Для защиты от накрутки добавлен `SpamGuard`. Один пользователь или одна сессия может засчитать один и тот же запрос максимум 3 раза за 1 минуту

Чтобы память не росла бесконечно, старые поисковые события удаляются при пересчете топа, а старые данные `SpamGuard` очищаются отдельной фоновой задачей

`TopCache` заранее считает Top-100. Если клиент запросил `limit=3`, сервис просто берет первые 3 элемента из уже готового списка. Иногда считается больше элементов, чем нужно, но API быстрее отвечает на запросы

Если `created_at` не передан, сервис использует время обработки события. Это удобно для локальной проверки
